About Go

Go is a new somewhat new-ish language developed by engineers at Google.

It was intended to be a systems language--a modern version of C. It’s also attracted a lot of interest from Ruby and Python developers.

Go has:

  • C-style syntax (curly braces define blocks)
  • static typing (often, automatically inferred)
  • garbage collection
  • modern data types like slices and maps (i.e. lists and dictionaries) and support for functional programming
  • not quite OOP: no objects, but structs are similar. implicit interfaces
  • a respectable (in some cases awesome) standard library for common tasks
  • full compilation to native binaries (no interpreter needed), and very quick compilation


It also has support for concurrent programming, in the form of goroutines (I think of these as very lightweight threads, but this is not technically correct) and channels w/ “select”.

First class support for concurrency is the hook that gets a lot of people interested in Go (including yours truly).

More on concurrency in Go later.


Getting Started

Getting started with Go is super easy:

http://tour.golang.org/

This is an interactive tutorial where you write Go code in your browser, you submit it to the server, and the server sends back the results of your compiled code. Pretty spiffy.

Noteworthy things that should be highlighted as part of examples, but not actually displayed as a second list:

  • Implicitly fulfilled interfaces (static ducks) (methods only)
  • Multiple return values from functions
  • Explicit error checking (no try/catch) (panic)
  • Only one looping construct (for)
  • Defer
  • Gofmt
  • No generics (magical append, though)
  • Pointers




Why you want to use Go

  • All the cool kids are using it, it's different but not TOO different
  • Quick and easy enough to use as your go-to scripting language
    (go-to . . . ahahaha)
  • Concurrency is cool (you’ll have to get your head around it eventually)
  • Net/http and other power tools in the standard library
  • High performance (at least compared to interpreted languages)
  • Compiled binaries are nice for deployment

Why you don't want to use Go

  • No exceptions
  • No generics (interface{} + reflection, bleah) (*)
  • Package management can get messy (no versioning) (*)
  • Idiosyncratic sometimes means annoying (import use, awkward string ops, error handling)
  • You don't like minimalist/TOOWTDI environments
